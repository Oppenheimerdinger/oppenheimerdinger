# oppenheimerdinger v0.3 "new-project" — 설계 스펙

**날짜**: 2026-07-14
**상태**: 사용자 승인 대기
**상위**: `2026-07-14-oppenheimerdinger-design.md` (우산)
**확정 결정 (2026-07-14)**: 스크립트+프로즈 하이브리드 / 커맨드 전용 발화 +
way-of-working 라우터 행 / hosts 풀 기계(manifest+setup.sh) / CLAUDE.md는
프로젝트 사실만.

## 범위

- `commands/new-project.md` — 인터뷰(질문+추천 기본값)를 진행하고 답을 플래그로
  모아 스크립트를 호출하는 커맨드 프로즈. 스킬 없음 (way-of-working 라우팅
  표에 "새 연구 프로젝트 시작 → `/new-project`" 행 추가로 발견성 해결).
- `assets/new-project.sh` — 결정론적 스캐폴더 (모든 입력은 플래그; 인터랙션
  없음; CI 스모크 가능).
- `assets/hosts-templates/` — manifest, setup.sh, README 템플릿 (외부코드 반입).
- `assets/CLAUDE.md.template` — 프로젝트 사실 전용 템플릿.
- `tests/new-project-smoke.sh` + CI.
- 제외 (명시): ohd-share 실행 스킬(백로그 유지), 다중 hosts 동시 스캐폴드
  (v0.3은 최대 1개 — 추가 반입은 드롭인 가이드/수동), GitHub org 고급 설정.

## 인터뷰 (커맨드 프로즈가 진행 — 전 항목 질문 + 추천 1개, 엔터면 기본값)

| # | 질문 | 플래그 | 기본값 (추천) |
|---|---|---|---|
| 1 | 프로젝트 이름/경로 | `--name`, `--dir` | dir = `~/projects/<name>` (사용자 관례 감지 시 그쪽) |
| 2 | GitHub repo 만들까? | `--github <owner/name\|none>` | 만들기, **private** (공개는 명시 요청 시) |
| 3 | 트렁크 브랜치명 | `--trunk` | `main` |
| 4 | 캠페인 명명 | `--naming free\|numbered` | free |
| 5 | 병합 모델 | `--merge-model coordinator\|review-gate` | 1인=coordinator, 협업=review-gate |
| 6 | 트렁크 보호 훅 (docs-only)? | `--hook\|--no-hook` | 협업/공유 repo면 hook, 개인 실험 repo면 no-hook |
| 7 | 외부 코드 반입 | `--host-name/--host-vehicle fork\|patches/--host-repo/--host-trunk` (전부 주거나 전부 생략) | 없음. "수정할 외부코드"면 fork, "패치 몇 장"이면 patches |
| 8 | **노드 토폴로지**: 한 머신에서만 도나, 원격 계산 노드(GPU 서버 등)가 있나? 있으면 각각의 이름/역할/접속법, **파일시스템 공유 여부** | `--node <name:role[:ssh-alias]>` (반복 가능), `--fs shared\|separate` | 단일 머신. 다중이면 fs=separate 가정(코드는 git-only 이동 — way-of-working 규율의 인스턴스) |
| 9 | **실행 환경(들)**: 환경이 하나인가, 프로그램/스테이지별로 갈리나? (예: 빌드용 conda + 분석용 uv + 노드별 module) 각 환경의 이름/타입/소속 머신 | `--env <name:uv\|conda\|module>[@machine]` (반복 가능) | `main:uv` 하나. 다중이면 primary 1개 지정 |
| 10 | 대용량 공유 데이터 | `--data-dir <path\|none>` | none (있으면 read-only symlink 패턴 기록) |
| 11 | 배포 형태 | `--deploy none\|snapshot\|mirror` | none — **기록만** (실행 도구는 ohd-share 백로그) |

토폴로지/환경 답변의 스캐폴드 반영 (질문만 하고 버리지 않는다):
- CLAUDE.md 사실 블록에 **머신×환경 매트릭스** 섹션 생성 — 행마다
  머신(이름/역할/ssh/fs 공유 여부)과 그 위의 환경(이름/타입/활성화 방법 자리).
  다중 노드 + fs=separate면 "코드 이동은 commit→push→pull, scp 금지" 인스턴스
  문구와 원격 노드용 clone 안내 한 줄 포함.
- env 스텁: **primary 1개**는 루트에 (uv → `pyproject.toml`+`.python-version`;
  conda → `environment.yml`), 추가 conda env는 `envs/<name>.yml`, module 타입은
  파일 없이 매트릭스에 로드 커맨드 자리만. uv는 루트 1개만 허용(다중 uv는
  스크립트가 거부하고 안내).

인터뷰 원칙: 대상 사용자는 dev-weak — 각 질문에 "왜 묻는지" 한 줄 + 추천 1개.
모호한 답은 재질문. 최종 실행 전 **요약 게이트**: 모인 플래그 전체를 보여주고
go 승인 후 스크립트 실행 (deep-solve 게이트 문법).

## `assets/new-project.sh` 스캐폴드 산출물

모든 플래그 필수 아님 — 위 기본값 내장. `--github none`이면 로컬만.

1. `git init -b <trunk>` @ `<dir>` (+ 초기 커밋, `.gitignore` 기본: `.omc/`,
   `.superpowers/`, env 산출물).
2. `tools/campaign.sh` — v0.2 assets 템플릿 복사 + config 헤더를 답변으로 치환
   (TRUNK/NAMING/MERGE_MODEL/STATE_DIR + host 있으면 DEP_DIR/DEP_TRUNK/PIN_FILE)
   + `docs/campaigns/.gitkeep`.
3. `tools/install-hooks.sh` 복사; `--hook`이면 즉시 실행.
4. `CLAUDE.md` — 템플릿 인스턴스화 (아래 §템플릿).
5. `README.md` 스텁 (이름/goal placeholder).
6. `.claude/skills/.gitkeep` — 프로젝트 스킬 자리 관례.
7. env(들): primary uv → `pyproject.toml` 최소 + `.python-version`(3.12);
   primary conda → `environment.yml`; 추가 conda → `envs/<name>.yml`; module →
   기록만. `--node`/`--fs`/`--env` 반복 입력은 CLAUDE.md 머신×환경 매트릭스로.
8. host 지정 시: `hosts/<name>/{manifest,setup.sh,README.md}` 템플릿
   인스턴스화 (PIN은 빈 값 + "첫 setup 후 기입" 주석).
9. `--data-dir` 지정 시: `data/` → read-only symlink 생성 + CLAUDE.md 기록.
10. `--github owner/name`이면: `gh repo create --private --source=. --push`
    (gh 없거나 실패 시 정확한 수동 커맨드 출력하고 나머지는 완료 처리 — 부분
    실패가 전체를 무르지 않음).

종료 출력: 생성된 구조 요약 + 다음 스텝 3줄 (① 새 세션을 repo에서 열기 ②
way-of-working 스킬 참조 ③ 첫 캠페인 `tools/campaign.sh new <이름>`).

## `assets/hosts-templates/` (umbrella-proj 패턴 일반화)

- **manifest**: `HOST= / VEHICLE=fork|patches / REPO= / TRUNK= / PIN= /
  PATCHES=` (KEY=VALUE, setup.sh와 campaign.sh pin이 공동 소비).
- **setup.sh**: manifest 소스 → 대상 dir 결정(`<HOST>_DIR` env >
  `<repo 부모>/<HOST>` 기본) → **FRESH**: clone + checkout PIN (fork) 또는
  clone+`git apply` PATCHES (patches, 의도적으로 dirty 유지) / **EXISTING**:
  **adopt as-is, HEAD 절대 이동 금지** (병렬 세션의 캠페인 브랜치 보호 — 원본
  패턴의 핵심 안전규칙) / `--build`는 업스트림 빌드 스크립트에 위임(있으면).
- **README.md**: "what we add" 서술 자리.

## `assets/CLAUDE.md.template` (프로젝트 사실만 — 규율 중복 금지)

섹션: ① 프로젝트 한 줄 + goal placeholder ② **사실 블록** (trunk, 명명, 병합
모델, worktree root, env 선택, data 경로, deploy 선택, hosts 목록 — 전부
인터뷰 답으로 치환) ③ **포인터**: "작업 규율은 oppenheimerdinger 플러그인의
way-of-working/campaign-land/campaign-status 스킬이 담당; 라이프사이클은
`tools/campaign.sh`" ④ 메모리 배선 한 줄 (프로젝트 메모리 슬러그 자리) ⑤
"이 파일은 claude-md-sanity의 감사 대상 — 사실이 바뀌면 여기도" 자기유지 문구.
규율(worktree 규칙 등)은 넣지 않음 — 플러그인 미설치 세션 대비는 ③의 포인터
문구에 "플러그인 없으면 docs/campaign-dropin.md 참조" 한 줄로.

## 스모크 (tests/new-project-smoke.sh, CI 추가)

임시 dir에서 `--github none`으로 2회 실행:
1. 기본 프로필 (uv, hook, host 없음): 구조 assert (git repo, trunk명, CLAUDE.md
   치환 확인, campaign.sh 헤더 값, pyproject 존재) → **내부에서
   `tools/campaign.sh new t1` → worktree/상태문서 → `abort t1`** (통합: 스캐폴드
   결과물이 실제로 동작) → hook 검증: 트렁크에서 비-docs 파일 커밋 시도 → 거부
   assert.
2. 풀 프로필 (numbered, review-gate, no-hook, host fork, 다중 env(conda
   primary + 추가 conda + module), 다중 node + fs=separate, data-dir):
   hosts/ 파일 존재+치환, environment.yml + envs/<name>.yml, CLAUDE.md에
   머신×환경 매트릭스와 git-only 이동 문구 존재, data symlink, campaign.sh
   헤더에 DEP_* 반영, numbered 검증(`new badname` 거부), 다중 uv 거부 검증.

네트워크/gh 불요 (github none).
