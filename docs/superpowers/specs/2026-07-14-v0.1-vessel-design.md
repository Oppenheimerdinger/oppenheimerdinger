# oppenheimerdinger v0.1 "그릇+이식" — 설계 스펙

**날짜**: 2026-07-14
**상태**: 사용자 승인 대기
**상위 문서**: `2026-07-14-oppenheimerdinger-design.md` (우산 스펙)
**리뷰 이력**: fable 이중 렌즈 리뷰(완결성/아키텍처) + 자체 점검 삼각측량 완료.
아키텍처 리뷰의 "플러그인 2개" 대안은 검토 후 기각(사용자 결정 — 실청중이

플러그인 그릇 + 검증된 기존 자산의 이식. 신규 설계는 `ohd-setup` 하나.

- 산출물: 설치 가능한 `oppenheimerdinger` v0.1.0 (공개 GitHub, MIT)
- 포함: deep-solve v0.2.2 흡수, claude-md-sanity·review-to-convergence 승격,
  `/ohd-setup`, 마이그레이션 완결
- 제외 (이후 슬라이스): way-of-working + campaign-land/status (**v0.2**),
  new-project (**v0.3**) — 순서는 리뷰 권고로 스왑 확정 (스캐폴더는 land 의식이
  존재하는 세상에 태어나야 함)

## `ohd-setup` (신규 — 커맨드 전용)

동작 순서:
1. `claude plugin list`로 점검: **superpowers** — "권장 (v0.2부터 하네스 필수;
   v0.1은 없어도 동작)" 라벨 정직하게. **oh-my-claudecode** — "선택 (ralph 쓸
   때만)". **잔존 `deep-solve@dipark`** — 발견 시 이중 등록 경고 + uninstall
   커맨드 제시.
2. 미설치 항목: marketplace add → install 순서의 정확한 커맨드를 제시하고
   **"내가 대신 실행할까?"를 항목별로 물어 승인 시 실행** (인터뷰 톤 그대로).
3. Workflow 툴 가용성 확인 (deep-solve isolated 모드 요건; 없으면 grounded/수동
   폴백 안내).
4. 결과를 체크리스트로 보고. **마지막 줄 고정**: "새로 설치했다면 세션 재시작
   (또는 /reload-plugins) 후 /ohd-setup을 다시 실행해 확인하세요" — 재시작
   없이는 등록 안 되는 함정이 dev-weak 사용자의 1번 좌초 지점.
5. 기능별 의존 매트릭스 형식: "X는 Y에 필요; 없으면 Z로 강등" (과장 금지 —
   신뢰 비용 방지).

## 이식 명세

### deep-solve (v0.2.2 → 여기)

- 파일 복사 (히스토리는 archive된 옛 repo가 보존). `.omc/`·`.superpowers/`류
  상태 디렉토리 제외.
- 커맨드:스킬 동명 충돌 workaround(@인라인) **명시적으로 유지** — 순진한
  복사로 누락 금지 (커맨드 파일 내 경고 주석도 그대로).
- SKILL.md 문구 갱신 1건: "review-to-convergence (if installed)" → 같은
  플러그인의 형제 스킬이므로 "the review-to-convergence skill (bundled)" 로.
- 버전 매핑 명문화 (README/CHANGELOG): "oppenheimerdinger 0.1.0은 deep-solve
  v0.2.2를 그대로 포함; 이전 히스토리는 archived repo 링크".
- README에 **제거/롤백 단락**: uninstall 커맨드 + 특정 버전 고정/다운그레이드
  방법 (릴리스가 문제일 때의 탈출구 — 대상 사용자용).
- **열린 백로그 이관** (`docs/backlog.md` + 이식된 spec에서 링크):
  ① SYNTH 후보 선정이 spec("계보별 최선")과 달리 전역 findings-top-2 —
  수정안: lineage id + 계보별 최선 + 테스트 1 ② 확증 불일치 결론이 COLD
  pitfall 목록으로 새는 경미한 앵커링 채널.

### 스킬 승격 2건

- `review-to-convergence`: 본문에 가드 한 줄 추가: "**고정된 산출물에만 사용
  — 리뷰가 산출물의 재설계를 유발하기 시작하면(움직이는 과녁) 루프를 멈추고
  설계 단계로 돌아가라.** 고정 산출물은 보통 1~3회에 수렴한다; 횟수 상한은
  두지 않되(메인 세션이 리뷰 내용을 전부 보므로 스스로 판단), 이례적으로
  길어지면 산출물 자체를 의심하라." 나머지 본문 그대로 + 상호참조 스윕.
- `claude-md-sanity`: **전신 일반화** (트리거만이 아니라 본문 — the-company/
  pkg-proj "두 곳!" 사례/cloud-gpubox<bigfs> 경로 전부). 회사 사례는 일반 예시로
  치환하되 이중언어(영/한) 감사 원칙과 방법론은 유지. `claude-md-improver`
  참조는 "if you have a rewriting tool" 일반 표현으로.
- 상호참조 스윕: 승격 스킬 텍스트 내 스킬명 참조 전수 점검 (개인
  delegating-hard-problems의 "REQUIRED COMPANION: review-to-convergence"는
  bare-name 매치라 생존 — 확인만).

### 공개 릴리스 게이트 (기계적)

```
grep -rniE "the-company|internal-|validation-proj|gpubox|<bigfs>|dipark" <shipped tree>
grep -rnE  "Oppenheimerdinger/deep-solve|deep-solve@dipark|deep-solve:deep-solve" <shipped tree>
```
1번 grep은 화이트리스트 3건(marketplace 이름 `dipark`, plugin.json author,
README 설치 커맨드의 `@dipark`)을 제외하고 0건이어야 push — 결과를 눈으로
훑어 화이트리스트 외 히트가 있으면 차단. 2번 grep은 docs/backlog.md의 의도적
역사 언급 제외 0건.

## 마이그레이션 절차 (순서 고정 — 리허설 후 실행)

1. **v0.1 완성** → 테스트 20/20 + plugin-dev validator/skill-reviewer 각 1패스
   + 릴리스 게이트 grep.
2. **GitHub 공개 push** (`Oppenheimerdinger/oppenheimerdinger`) → **신선 설치
   end-to-end 검증** (marketplace add → install → 새 세션 → /ohd-setup →
   /deep-solve 스모크).
3. **이주 리허설 (내 머신 1대)**: 기존 `dipark` marketplace(옛 repo행)가 있는
   상태에서 remove → add → uninstall deep-solve@dipark → install → 새 세션
   가시성. add-before-remove 실패 모드 관찰·기록.
4. **옛 repo 마지막 커밋**: README 최상단 "moved" + 정확한 4줄 이주 레시피,
   deep-solve SKILL description에도 이주 문구, 버전 bump (기설치자의
   `claude plugin update`가 이주 안내를 보게 됨). push 후 **GitHub archive**
   (archive 후엔 push 불가 — 순서 엄수).
5. **내 머신들 순회**: 세션 노드, cloud-gpubox, gpu-server-b — 각각 marketplace
   remove→add, uninstall/install, 개인 스킬 2종 백업 후 제거
   (`~/.claude/skills-backup-<date>/`로 이동; rm 금지), 세션 재시작.
6. **메모리/문서 갱신**: deep-solve-plugin.md를 oppenheimerdinger-plugin.md로
   통합(새 경로·커맨드·백로그 포인터), MEMORY.md 인덱스 수정.
7. **동료 재공유**: 갱신된 한국어 가이드 + "예전 링크는 이제 안내용" 명시를
   같은 채널에.

## Repo 자체 개발 규약 (CLAUDE.md에 명문화)

- **RELEASING 리추얼**: plugin.json 버전 bump → 테스트 → 커밋·태그 → push →
  `claude plugin update oppenheimerdinger@dipark` → 세션 재시작/reload → 확인.
  (버전 게이트 캐시: bump 없으면 변경이 배포 안 됨 — 1번 함정.)
- 테스트 호출 형식: `node --test tests/*.test.mjs` (디렉토리 형식은 일부 Node
  에서 실패).
- 스킬 description 예산 ~50단어; `ohd-` 커맨드 접두어 관례.
- **lock-step (두 곳!)**: superpowers 의존 라벨("v0.1 권장 → v0.2부터 필수")은
  ohd-setup.md와 (v0.2의) way-of-working 두 곳에 존재 — 한쪽 변경 시 반드시
  같이. md-sanity의 감사 대상.
- v0.1은 main 직행 (전용 repo); campaign 셀프호스팅 여부는 v0.2에서 재검토.
